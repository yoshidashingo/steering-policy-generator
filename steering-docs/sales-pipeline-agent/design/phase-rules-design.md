# Phase Rules Design

## Phase Rule Files 一覧

### Phase 1: SURVEILLANCE（4ファイル）

#### S1: `surveillance/schedule-check.md`
- **Classification**: ALWAYS
- **Purpose**: 巡回タイミング判定（日中/夜間モード切り替え）、前回サイクル完了確認、Salesforce API使用量確認
- **Content Outline**:
  - 時間帯判定ロジック: JST 8:00-20:00 = 日中モード、20:00-8:00および祝日 = 夜間モード
  - 祝日カレンダー参照（日本祝日 + 会社カレンダー設定ファイル参照）
  - 前回サイクルの正常完了確認: 前回サイクルが45分タイムアウトで強制終了していた場合、繰り延べリストを引き継ぐ
  - Salesforce API使用量取得・判定:
    - 使用率 < 80%: 通常モード（1時間巡回）
    - 使用率 80-95%: セーフモード移行（巡回間隔を2時間に延長、管理者Slack通知）
    - 使用率 >= 95%: 読み取り専用モード（DML操作停止、巡回のみ続行）
  - セーフモード中の承認済みアクション: Salesforceへの書き込みを保留し次サイクルへ繰り延べ
  - 夜間モード時のダイレクトパス: S2→S3→S4をスキップしてR2（Next-Day Prep）へジャンプ
- **Domain Injection Points**: 時間帯定義（JST、祝日カレンダー）、APIガバナーリミット閾値、セーフモード切り替え条件
- **GOOD例**: API使用率が83%に達したためセーフモードに移行し、巡回間隔を2時間に延長。ガバナーリミット到達を回避した
- **BAD例**: 前回サイクルが45分タイムアウトで終了していたにもかかわらず繰り延べリストをリセットし、未対応アクションが消失した（Pitfall 4対策漏れ）
- **Est. Lines**: 150-200

---

#### S2: `surveillance/source-connection.md`
- **Classification**: ALWAYS
- **Purpose**: 4ソース（Salesforce/Gmail/Google Calendar/Slack）の接続確認・OAuth認証・トークンリフレッシュ
- **Content Outline**:
  - 各ソースの接続確認シーケンス（並行実行可）
  - OAuthアクセストークン有効期限チェック: 残り30分以内の場合はリフレッシュトークンで更新
  - トークンリフレッシュ失敗時: 当該ソースを「障害」フラグで記録し次のステージへ続行（フォールバック）
  - 部分障害時の続行判断:
    - Salesforceのみ障害: 承認制アクションを停止し読み取り専用で続行
    - Gmail/Calendar/Slackのみ障害: 当該ソース由来の検出カテゴリをスキップして続行
    - 全ソース障害: 巡回をスキップし障害通知を送信
  - 接続状態をサイクル状態ファイルに記録
- **Domain Injection Points**: 4ソースのOAuthスコープ定義（data-source-config.md参照）、フォールバック優先順位
- **GOOD例**: Gmailのアクセストークンが期限切れだったため自動リフレッシュし、他3ソースで巡回を続行。未返信検出カテゴリのみスキップとレポートに明記した
- **BAD例**: Salesforce接続失敗をエラーとして処理を中断した。他ソースで取得可能なデータがあるにもかかわらず巡回全体が停止した（Pitfall 4対策漏れ）
- **Est. Lines**: 150-200

---

#### S3: `surveillance/data-collection.md`
- **Classification**: ALWAYS
- **Purpose**: 4ソースからの差分データ収集、統合ビュー構築、ソース間整合性チェック
- **Content Outline**:
  - 差分取得: 前回巡回タイムスタンプ以降の変更データのみ取得（全量取得禁止）
  - Salesforce収集項目: 商談（Opportunity）ステージ・金額・クロージング日・最終更新日、Activity履歴、リード一覧
  - Gmail収集項目: 送受信メール（Subject/From/To/Date/Snippet）、スレッドID
  - Google Calendar収集項目: 外部参加者を含むミーティング予定（今後14日）
  - Slack収集項目: 指定チャンネルの商談名・顧客名を含むメッセージ
  - 統合ビュー構築: タイムスタンプベースで最新情報を優先にマージ
  - ソース間矛盾検出（Pitfall 5対策）:
    - Salesforceステージ「Closed」 かつ Gmailで直近7日以内に商談継続メール → 矛盾フラグ付与
    - Salesforce Activity未記録 かつ Gmail/Calendar/Slackでアクティビティあり → ステータス未反映フラグ
    - 矛盾フラグ付き案件はASSESSMENTで人間判断を仰ぐ（自動判定禁止）
  - Salesforce Bulk API 2.0使用: 50件以上のレコード取得はバルクAPIで一括取得（Pitfall 4対策）
- **Domain Injection Points**: 各ソースのAPI呼び出しパラメータ、矛盾検出パターン定義、Bulk API閾値
- **GOOD例**: Salesforceで停滞判定対象の商談Xについて、Slackで担当者が2日前にやり取りをしていることを検出。矛盾フラグを付与し、自動停滞アラートを保留にしてAssessment時に担当者に確認を委ねた（Pitfall 5対策）
- **BAD例**: 統合ビュー構築時にSalesforceのステージデータのみを参照し、Slackでの活発なやり取りを無視して停滞商談と誤判定した（Pitfall 1対策漏れ）
- **Est. Lines**: 200-260

---

#### S4: `surveillance/change-detection.md`
- **Classification**: ALWAYS
- **Purpose**: 統合ビューから7種の変更・異常を検出
- **Content Outline**:
  - 検出ルール1 — 停滞商談（Pitfall 1対策: 多層判定必須）:
    - Salesforceで閾値超え（Proposalステージ14日、Negotiationステージ7日、その他ステージ21日）
    - **かつ** Gmail/Slackで直近N日以内にアクティビティなし（Nは閾値の半分）
    - 矛盾フラグ付き案件は自動検出対象外（人間判断に委ねる）
  - 検出ルール2 — 未返信メール:
    - 担当者送信メールへの返信が3営業日超過
    - すでにリマインダー送信済みの場合は重複通知抑制（Pitfall 3対策）
  - 検出ルール3 — 会議準備:
    - Google Calendarで外部参加者ありの予定が24時間以内
    - かつ対応するSalesforce商談が存在する
  - 検出ルール4 — 金額変動:
    - SalesforceのOpportunity金額フィールドが前回巡回比で10%以上変動
  - 検出ルール5 — 期限接近:
    - クロージング日まで7日以内（緊急: 1日以内）
    - すでに通知済みの場合は48時間以内の重複通知抑制（Pitfall 3対策）
  - 検出ルール6 — 未追跡リード:
    - Gmail/CalendarのやりとりでSalesforceにリード/商談が存在しない外部ドメイン
  - 検出ルール7 — ステータス未反映:
    - Gmail/Calendar/SlackでActivityがあるが、SalesforceのActivityログに記録なし
  - 検出結果の優先度分類: 緊急（期限1日以内・大型商談の異変）/ 重要 / 情報
  - 同一商談への複数ルール同時適用時の統合処理（1商談1アクション原則）
- **Domain Injection Points**: ステージ別滞留閾値、多層判定のアクティビティ対象期間、重複通知抑制間隔
- **GOOD例**: 商談Aが「Proposalステージ16日」でSalesforceでは停滞判定対象だが、Slack#deal-updatesで担当者が顧客と交渉中のメッセージを検出。停滞ではなく「ステータス未反映」として検出した（Pitfall 1・Pitfall 5対策）
- **BAD例**: 同一商談に「停滞」「未返信」「期限接近」の3つの検出結果を個別に通知し、担当者が同日に3件の通知を受け取った（Pitfall 3対策漏れ）
- **Est. Lines**: 220-280

### Phase 2: ASSESSMENT（4ファイル）

#### A1: `assessment/deal-health-scoring.md`
- **Classification**: ALWAYS
- **Purpose**: 商談ヘルススコアの算出（停滞度・エンゲージメント・期限リスクの複合評価）
- **Content Outline**:
  - スコア計算式:
    ```
    HealthScore = (EngagementScore × 0.4) + (StageProgressScore × 0.3) + (DeadlineScore × 0.3)
    EngagementScore: 最終アクティビティ経過日数・メール応答率・ミーティング頻度から算出
    StageProgressScore: ステージ進行速度・MEDDIC/BANT充足率から算出
    DeadlineScore: クロージング日までの残日数・現ステージの平均クローズ日数から算出
    ```
  - スコアレンジ: 赤（0-39）/ 黄（40-69）/ 緑（70-100）
  - **多層補正（Pitfall 5対策）**: Salesforceスコアが低くても、Gmail/Slackでの直近アクティビティがある場合はEngagementScoreを補正
  - ヘルススコアはサイクル状態ファイルに記録し、前回比較でトレンドを算出
  - 加重パイプライン算出: ステージ別受注確率（Prospecting 10% / Qualification 25% / Proposal 50% / Negotiation 75% / Closed Won 100%）× 商談金額
- **Domain Injection Points**: ステージ別受注確率テーブル、アクティビティ経過日数閾値、MEDDIC/BANT項目定義
- **GOOD例**: SalesforceのステージProgressScoreは低かったが、GmailとSlackのEngagement補正により総合スコアが黄から緑に修正された。誤った停滞アラートを回避した（Pitfall 5対策）
- **BAD例**: Salesforceの最終更新日のみでスコアを算出し、担当者がSlackで活発に交渉中の商談を赤（要緊急対応）と誤判定した（Pitfall 1対策漏れ）
- **Est. Lines**: 180-230

---

#### A2: `assessment/action-identification.md`
- **Classification**: ALWAYS
- **Purpose**: S4の検出結果とヘルススコアから具体的な対応アクションを生成
- **Content Outline**:
  - 検出種別ごとのアクション生成ルール:
    | 検出種別 | 生成アクション |
    |---------|--------------|
    | 停滞商談 | ステータス確認リマインダー送信（担当者）+ Salesforce Activity記録（自動） |
    | 未返信メール | フォローアップメール送信（承認制）+ Activity記録（自動） |
    | 会議準備 | 会議準備サマリー生成・送付（自動） |
    | 金額変動 | 金額変更確認通知（担当者、承認制） |
    | 期限接近 | クロージング期限アラート（担当者、自動）+ 必要ならディール加速提案（承認制） |
    | 未追跡リード | リード追加提案（承認制） |
    | ステータス未反映 | Activity自動記録（自動）+ ステージ更新提案（承認制） |
  - 矛盾フラグ付き案件: アクション生成を保留し「人間確認依頼」アクションを生成
  - 1商談1アクション原則: 同一商談に複数の検出がある場合、最高優先度のアクションのみ生成（Pitfall 3対策）
  - アクションのコンテキスト付与: 「商談X（金額¥5M、クロージング予定4/30）がProposalステージで14日間停滞。最後のActivityは3/20のメール送信」
- **Domain Injection Points**: 検出種別とアクションのマッピングテーブル、コンテキスト生成テンプレート
- **GOOD例**: 商談Bに「未返信」「期限接近」「金額変動」の3つの検出があったため、最高優先度の「期限接近」アクション1件のみ生成し、他の情報はコンテキストとして付加した（Pitfall 3対策）
- **BAD例**: 1商談に3件のアクション（未返信リマインダー + 期限アラート + 金額確認）を個別に生成し、担当者が同一商談について3件の承認リクエストを受け取った（Pitfall 3対策漏れ）
- **Est. Lines**: 180-230

---

#### A3: `assessment/priority-ranking.md`
- **Classification**: ALWAYS
- **Purpose**: 対応アクションの優先度スコアリングと実行順序の確定
- **Content Outline**:
  - 優先度スコア計算式:
    ```
    PriorityScore = (DealAmount / 1,000,000) × AmountWeight
                  + (1 / DaysToClose) × DeadlineWeight
                  + (1 - HealthScore / 100) × UrgencyWeight
    AmountWeight=0.3, DeadlineWeight=0.4, UrgencyWeight=0.3
    ```
  - 優先度レベル:
    - P1（Critical）: スコア上位10%、またはクロージング日1日以内
    - P2（High）: スコア上位30%、またはクロージング日7日以内
    - P3（Medium）: スコア上位60%
    - P4（Low）: それ以外
  - P1アクションは同一サイクル内で必ず処理（ループ繰り延べ対象外）
  - 大型商談（¥10M以上）はAmountWeightを1.5倍で計算
  - タイムアウトリスク: サイクル残り時間が15分以下の場合はP1・P2アクションのみ処理し、P3・P4は次サイクルへ繰り延べ
- **Domain Injection Points**: 商談金額の大型定義（¥10M閾値は設定可能）、P1必ず処理の例外定義
- **GOOD例**: 残り時間12分でP3・P4アクションを次サイクルに繰り延べ、P1（クロージング明日の¥8M商談）の承認リクエストを確実に送信した
- **BAD例**: 優先度を考慮せずFIFO順でアクションを処理し、タイムアウト直前に大型商談の期限接近アラートが処理されなかった
- **Est. Lines**: 160-210

---

#### A4: `assessment/authority-classification.md`
- **Classification**: ALWAYS
- **Purpose**: 各アクションをaction-authority.mdの基準に基づいて自動実行/承認制に分類
- **Content Outline**:
  - action-authority.mdの低リスク/高リスクテーブルを参照して各アクションを分類
  - 分類の実行:
    - 自動実行リスト: D1（Auto-Execute）で処理
    - 承認制リスト: D2（Approval Request）+ D3（Approval Processing）で処理
    - 保留リスト（矛盾フラグ付き）: 担当者確認依頼として承認制に追加
  - 境界曖昧ケースの保守的判定: どちらか判断できない場合は承認制として扱う（Pitfall 2対策）
  - 分類理由のログ記録: なぜ自動実行/承認制と判断したかを監査ログに記録
  - 承認リクエストのバッチ化（Pitfall 3対策）: 同一担当者への承認リクエストは最大3件ずつバッチにまとめる
- **Domain Injection Points**: action-authority.mdの低リスク/高リスクテーブル参照、境界曖昧ケースの定義
- **GOOD例**: ステージ更新を「Prospecting→Qualification（前進）」と判断して自動実行とした。一方「Proposal→Qualification（後退）」は高リスクとして承認制に分類した（Pitfall 2対策）
- **BAD例**: 「ステージ更新」というアクション種別だけを見て前進/後退を区別せず一律に承認制にした。毎サイクル10件の承認リクエストが発生し担当者が対応を放棄した（Pitfall 3対策漏れ）
- **Est. Lines**: 150-200

### Phase 3: DISPATCH（5ファイル）

#### D1: `dispatch/auto-execute.md`
- **Classification**: CONDITIONAL（自動実行対象がある場合）
- **Execute IF**: A4の自動実行リストにアクションが1件以上ある場合
- **Skip IF**: 自動実行対象が0件の場合
- **Purpose**: 低リスクアクションの安全な自動実行
- **Content Outline**:
  - 実行前の排他制御: 同一Salesforceレコード（Opportunity ID）へのロック取得
  - Salesforce API呼び出しの最適化: バルクAPI 2.0で複数レコードを一括更新（個別更新の禁止）
  - 実行前スナップショット: 更新前のSalesforceレコード状態をキャプチャ（D5のロールバック用）
  - Activity自動記録テンプレート: 「[エージェント記録] [日時] Gmail送受信/Calendarミーティング/Slack言及を検出。アクティビティとして記録。」
  - Gmail/Slack通知の実行: 内部リマインダー、会議準備サマリーの送信
  - 実行完了記録: アクションID・対象レコードID・実行時刻を監査ログに記録
  - エラー時の処理: 個別アクションの失敗は他のアクション実行に影響させない（部分成功を許容）
- **Domain Injection Points**: Salesforce Bulk API 2.0の呼び出しパターン、Activityログ記録テンプレート、排他ロック取得方法
- **GOOD例**: 5件のActivity自動記録をBulk APIで一括送信し、API呼び出しを5回から1回に削減した（Pitfall 4対策）
- **BAD例**: 同一商談に対してSalesforce ActivityをAPIで記録中に、別スレッドが同じ商談のステージを更新し、競合状態でデータが破損した（排他制御漏れ）
- **Est. Lines**: 180-230

---

#### D2: `dispatch/approval-request.md`
- **Classification**: CONDITIONAL（承認制対象がある場合）
- **Execute IF**: A4の承認制リストにアクションが1件以上ある場合
- **Skip IF**: 承認制対象が0件の場合
- **Purpose**: 高リスクアクションの承認リクエスト送信
- **Content Outline**:
  - 承認リクエストのフォーマット:
    ```
    【承認リクエスト】[商談名]
    推奨アクション: [アクション種別]
    理由: [検出根拠 + コンテキスト情報]
    商談詳細: 金額¥[X]M / ステージ[Y] / クロージング予定[Z]
    ↓ 承認または却下を選択してください
    [承認する] [却下する（理由を入力）]
    ```
  - 送信チャンネル: 担当者SlackDM（プライマリ）、メール（フォールバック）
  - バッチ化（Pitfall 3対策）: 同一担当者への承認リクエストは最大3件を1メッセージにまとめる
  - リクエストID発行: 各承認リクエストに一意のIDを付与し、D3での結果照合に使用
  - タイムアウト管理: 送信時刻をサイクル状態ファイルに記録し、30分・2時間の管理をD4で実施
  - 夜間モード中の保留: 承認済みアクションでも顧客向け送信は翌営業日8:00まで保留
- **Domain Injection Points**: 承認リクエストのSlackメッセージフォーマット、バッチサイズ設定（最大3件）、送信チャンネル設定
- **GOOD例**: 1つの商談に対して「フォローアップメール送信」「ステージ更新提案」「リード商談化」の3件の承認リクエストを1件のSlackメッセージにまとめて送信した（Pitfall 3対策）
- **BAD例**: 10件の承認リクエストを個別に1件ずつSlack DM送信し、担当者のDMが承認リクエストで埋め尽くされた（Pitfall 3対策漏れ）
- **Est. Lines**: 180-230

---

#### D3: `dispatch/approval-processing.md`
- **Classification**: CONDITIONAL（承認済みアクションがある場合）
- **Execute IF**: 承認リクエストへの返答（承認/却下）が存在する場合
- **Skip IF**: 承認待ちアクションが全て未応答の場合
- **Purpose**: 承認結果の取得・実行・却下処理
- **Content Outline**:
  - 承認結果の取得: Slack/メールのインターアクティビティから承認/却下を判定
  - 承認時の実行:
    - D1と同様の排他制御・スナップショット取得を適用
    - 夜間モード中の顧客向けアクション: 承認済みでも実行を翌8:00まで保留
    - Salesforce書き込みはセーフモード・読み取り専用モードの制約に従う（Pitfall 4対策）
  - 却下時の処理:
    - 却下理由をサイクル状態ファイルに記録
    - 同一パターンの却下が3回蓄積された場合: 自動判定ロジックの見直しフラグを立て、管理者に通知
    - 却下されたアクションは同サイクルで再提案しない（Pitfall 3対策）
  - 実行完了記録: 承認者・承認時刻・実行結果を監査ログに記録
- **Domain Injection Points**: 承認結果の取得方法（Slackインターアクティビティ設定）、却下理由の分類定義
- **GOOD例**: 「商談ステージをProposal→Negotiationに更新」が承認されたため、実行前にSalesforceの現状スナップショットを取得してから更新した。D5でのロールバック準備が完了した
- **BAD例**: 承認された「顧客へのフォローアップメール送信」を夜間22:00に即時実行した。顧客が深夜にメールを受信し不快感を示した（夜間モード制御漏れ）
- **Est. Lines**: 160-210

#### D4: `dispatch/loop-control.md`
- **Classification**: ALWAYS
- **Purpose**: 残件チェック、ループ上限・タイムアウト判定、次ループまたは終了の決定
- **Content Outline**:
  - 残件確認: 自動実行リスト・承認待ちリストの未処理件数チェック
  - ループ継続条件:
    - 残件あり **かつ** ループカウント < 5 **かつ** 経過時間 < 45分 → A1（Deal Health Scoring）に戻る
  - ループ終了条件:
    - 残件0 → D5（Rollback Check）へ
    - ループカウント >= 5 → D5へ（残件を次サイクルに繰り延べ）
    - 経過時間 >= 45分 → D5へ（残件を次サイクルに繰り延べ、タイムアウトフラグ）
  - 新規検出の繰り延べ: ループ中に次サイクル繰り延べリストに追加された案件は、現サイクルのアクションリストに追加しない（Pitfall 3対策: 通知爆発防止）
  - 承認タイムアウト管理:
    - 承認リクエスト送信から30分経過: Slack DM自動リマインダー送信
    - 承認リクエスト送信から2時間経過: 未承認を「繰り延べ」として次サイクルのキューに追加
  - ループカウント・経過時間をサイクル状態ファイルに更新
- **Domain Injection Points**: ループ上限（5）、タイムアウト（45分）、承認リマインダー（30分）、承認繰り延べ（2時間）の設定値
- **GOOD例**: ループカウント5に達した時点で残件2件を次サイクルに繰り延べ、D5でロールバックチェックを実施してサイクルを正常完了した
- **BAD例**: 承認待ちのアクションが解決するまで無限にループし、次の巡回タイミングを超過してスケジュールが乱れた（ループ制御なし）
- **Est. Lines**: 160-210

#### D5: `dispatch/rollback-check.md`
- **Classification**: ALWAYS
- **Purpose**: 自動実行・承認済み実行の結果を検証し、異常を検出した場合にロールバックを実行
- **Content Outline**:
  - 検証対象: 本サイクルで実行した全自動実行・承認済み実行アクション
  - 検証方法: 実行前スナップショットと実行後のSalesforceレコード状態を比較
  - 異常検出パターン:
    - ステージが意図せず後退している（前進更新のつもりが後退）
    - 金額が意図しない値に変更されている
    - 削除すべきでないレコードが削除されている
  - ロールバック実行:
    - Salesforce Undelete API / レコード復元APIを使用
    - ロールバック完了後、担当者にSlack通知を送信
    - ロールバック失敗時: 管理者緊急通知 + 手動修正ガイド提示 + 該当アクション種別の自動実行を一時停止
  - ロールバック不要（正常）: 検証OK記録を監査ログに追記してR1へ進む
  - 異常なし・ロールバック不要の場合も検証完了記録は必須
- **Domain Injection Points**: ロールバック対象アクション種別、Salesforce復元APIの呼び出しパターン、一時停止の解除条件
- **GOOD例**: 「Proposalステージへの前進更新」を実行後、SalesforceレコードがQualificationに戻っていることを検出（前進のつもりが後退）。即座にロールバックし、担当者にSlackで「自動更新を取り消しました」と通知した
- **BAD例**: 自動実行完了後に実行結果を検証せず、誤ったステージ更新がそのまま残存した。翌日の巡回で発覚したが、顧客とのやり取りに支障が出ていた（Pitfall 2対策漏れ）
- **Est. Lines**: 160-210
