# コンプライアンスチェック — レビューフェーズ

## 目的

このステージでは、RFPのすべての要件が提案書で対応されているかを体系的に検証します。コンプライアンスチェックは、コンプライアンスマトリクスと実際の提案書内容を一行ずつ照合する監査であり、執筆中に漏れた要件がないこと、またすべてのコンプライアンス主張が裏付けられていることを確認します。

## 前提条件
- すべての前フェーズが完了していること（分析、計画、執筆）
- 執筆済みのすべての提案書セクションが利用可能であること：
  - `proposal-docs/proposal/executive-summary.md`
  - `proposal-docs/proposal/technical-response.md`
  - `proposal-docs/proposal/management-response.md`（実行された場合）
  - `proposal-docs/proposal/pricing-framework.md`（実行された場合）
- 元のコンプライアンスマトリクス：`proposal-docs/analysis/compliance-matrix.md`
- 共通ルールの読み込み：`common/content-validation.md`

## 実行分類
**タイプ**: 常時実行（ALWAYS）
コンプライアンスチェックは常に実行されます。非準拠の提案書を提出することは、すべての努力を無駄にすることになるためです。

---

## 実行ステップ

### ステップ1：読み込みとクロスリファレンス
**アクション**: コンプライアンスマトリクスとすべての提案書セクションを読み込む
**入力**: コンプライアンスマトリクス、すべての執筆済み提案書セクション
**出力**: 各要件のクロスリファレンスステータス：
- 要件ID
- 主張されたコンプライアンスステータス（マトリクスから）
- 対応している実際の提案書セクションおよび段落
- 検証ステータス（対応済み / 未対応 / 部分対応）
**検証**: マトリクス内のすべての要件が実際の提案書内容と照合されていること

### ステップ2：必須要件の監査
**アクション**: すべての必須要件が完全に対応されていることを検証する
**入力**: ステップ1のクロスリファレンス、要件文書（必須項目）
**出力**: 各必須要件について：
- 対応されているか？（対応済み / 未対応 / 部分対応）
- どこで対応されているか？（具体的なセクションおよび段落の参照）
- 回答は十分か？（完全準拠 / 部分準拠 / 不十分）
- 特定された問題点（詳細の欠如、整合性の不一致、矛盾）
**検証**: 未対応の必須要件がゼロであること。ギャップがある場合は重大な発見として扱う

### ステップ3：任意要件の監査
**アクション**: スコアを最大化するために任意要件が対応されていることを検証する
**入力**: ステップ1のクロスリファレンス、要件文書（任意項目）
**出力**: 各任意要件について：
- 対応されているか？（対応済み / 未対応）
- 対応済みの場合：回答の強度（強い / 十分 / 弱い）
- 未対応の場合：推定されるスコアへの影響
**検証**: 重みの大きい任意要件は対応されていること。重みの小さい項目は含める場合のコスト/ベネフィットを評価する

### ステップ4：提出形式の検証
**アクション**: 提案書がすべての形式要件および提出要件を満たしていることを検証する
**入力**: 要件文書からの提出要件、実際の提案書文書
**出力**: 形式準拠チェックリスト：
- ページ数とページ制限の対比
- 必要なセクションが存在し、正しい順序であること
- フォント、余白、書式の準拠（該当する場合）
- すべての必要な書類および証明書が含まれていること
- 巻構成が要件に合致していること
**検証**: すべての形式要件を満たしていること。逸脱がある場合は重大な発見としてフラグを立てる

### ステップ5：セクション間の整合性チェック
**アクション**: すべての提案書セクション間の整合性を検証する
**入力**: すべての提案書セクション
**出力**: 整合性に関する発見事項：
- 会社名およびブランディングが一貫していること
- チームメンバーの氏名および肩書がセクション間で一貫していること
- 技術用語および製品名が一貫していること
- タイムラインの日付が技術セクションとマネジメントセクション間で一貫していること
- 価格構成が技術スコープと整合していること
**検証**: セクション間に矛盾がないこと。用語が全体を通じて一貫していること

### ステップ6：コンプライアンスチェックレポートの生成
**アクション**: コンプライアンス検証レポートを作成する
**入力**: ステップ1〜5のすべての発見事項
**出力**: `proposal-docs/review/compliance-check-report.md`
**検証**: レポートが完全であり、発見事項を正確に反映し、アクション項目が明確に特定されていること

### レビュー必要 / REVIEW REQUIRED

**コンプライアンスチェックが完了しました。** 概要は以下の通りです：

- [N]件の要件を検証：[N]件が完全対応、[N]件が部分対応、[N]件が未対応
- [N]件の重大な発見事項（即座の対応が必要）
- [N]件の警告事項（改善を推奨）
- 形式準拠：[合格/不合格]
- セクション間整合性：[N]件の問題あり

### 次のステップ / WHAT'S NEXT

以下のいずれかを選択してください：

**A) 変更を依頼する** — 次に進む前にコンプライアンスの発見事項に対応します
**B) 品質レビューに進む** — コンテンツ品質評価に進みます

## エラーハンドリング

**エラー**: 重大なコンプライアンスギャップが見つかった場合（必須要件が未対応）
- **解決策**: 特定のギャップに対処するために執筆フェーズに戻る。ギャップの詳細と推奨コンテンツを提示する
- **次に進まないこと**: 未解決の重大なギャップがある状態で最終組立に進まないこと

**エラー**: 形式要件に違反している場合（ページ制限超過、構成の誤り）
- **解決策**: 具体的な違反事項を提示し、コンテンツの削減または再構成を提案する
- **復旧方法**: 優先度の低いコンテンツを削減するか、要件を満たすようにフォーマットを変更する

**エラー**: セクション間の矛盾が見つかった場合
- **解決策**: 矛盾を提示し、どちらのバージョンが正しいかをユーザーに確認する
- **復旧方法**: 影響を受けるセクションを更新し、整合性を維持する

一般的なエラーハンドリング手順については `common/error-handling.md` を参照してください。
