# 質問フォーマットガイド

## 目的

このファイルは、提案書作成ワークフロー中に行われるすべての質問の標準化フォーマットを定義します。一貫した質問フォーマットにより、明確なコミュニケーション、回答の検証、矛盾検出が可能になります。

---

## 基本ルール

### ルール1: 質問は専用ファイルに記載する
- チャットで直接質問しない
- すべての質問は構造化された質問ファイルとして生成する
- 質問ファイルは `proposal-docs/` 配下の適切なフェーズディレクトリに保存する

### ルール2: 多肢選択形式
- すべての質問は多肢選択形式（A, B, C, D オプション）を使用する
- 最後のオプションは必ず「その他（[Answer]: タグの後に記述してください）」とする
- 最低2つの意味のあるオプション + 「その他」オプション
- 最大4つの意味のあるオプション + 「その他」オプション

### ルール3: [Answer]: タグ
- すべての質問にオプションの後に `[Answer]:` タグを含める
- ユーザーは [Answer]: タグに記入して回答する
- 回答はレターコード（A, B, C）または「その他」後の自由記述

### ルール4: 回答の検証
- 回答を受け取った後、完全性と一貫性を検証する
- 矛盾が検出された場合、確認質問ファイルを生成する
- 矛盾する回答が未解決のまま進行しない

---

## 質問ファイルテンプレート

```markdown
# [フェーズ/ステージ名] 質問

以下の質問にお答えください。[目的に応じたコンテキスト説明]

## 質問 1
[明確で具体的な質問文]

A) [最初の意味のあるオプション]
B) [2番目の意味のあるオプション]
C) [3番目の意味のあるオプション]
D) その他（[Answer]: タグの後に記述してください）

[Answer]:

## 質問 2
[明確で具体的な質問文]

A) [最初の意味のあるオプション]
B) [2番目の意味のあるオプション]
C) [3番目の意味のあるオプション]
D) [4番目の意味のあるオプション]
E) その他（[Answer]: タグの後に記述してください）

[Answer]:
```

## 確認質問ファイルテンプレート

ユーザーの回答に矛盾や曖昧さが検出された場合：

```markdown
# [フェーズ/ステージ名] 確認質問

回答に[矛盾/曖昧さ]が検出されたため、確認が必要です：

## 問題 1: [簡潔な説明]
「[回答A]」（Q[X]:[レター]）と「[回答B]」（Q[Y]:[レター]）が示されました。
[矛盾/曖昧さの理由の説明]

### 確認質問 1
[問題を解決するための具体的な質問]

A) [最初の回答に向かう解決オプション]
B) [2番目の回答に向かう解決オプション]
C) [中間的な解決オプション]
D) その他（[Answer]: タグの後に記述してください）

[Answer]:
```

## フェーズ別の質問カテゴリ

### 分析フェーズの質問
- RFPのスコープと種類の明確化
- 曖昧な場合の要件解釈
- 矛盾する要件間の優先順位
- コンプライアンス能力の評価

### 計画フェーズの質問
- 競争環境とポジショニング
- 組織の強みと差別化要因
- コンテンツの優先順位と重点領域
- チーム構成と主要人員の可用性

### 執筆フェーズの質問
- 技術的アプローチの好み
- 参照する過去の実績事例
- 価格戦略と制約
- セクションの重点と詳細レベル

### レビューフェーズの質問
- コンプライアンスギャップの解決戦略
- ページ制限が厳しい場合の品質トレードオフ
- 最終フォーマットの好み
- 提出ロジスティクス

## 回答処理ルール

### ステップ1: 回答の収集
- ユーザーが記入した質問ファイルを読み込む
- [Answer]: タグを使って回答を抽出する
- レター回答をオプションテキストにマッピングする

### ステップ2: 完全性の検証
- すべての質問に回答があるか確認する
- 空白または不明確な回答を特定する
- 未回答の質問をフォローアップ用にフラグ付けする

### ステップ3: 矛盾の検出
提案書コンテキストでの一般的な矛盾パターン：
- 「積極的な価格設定」と「プレミアムサービスレベル」の両方を主張
- 「最小限のチーム」と「包括的なプロジェクト管理」の両方を選択
- 「迅速な納品」と「広範な品質保証」の両方を選択
- 「固定価格」と「スコープの柔軟性」の両方を示す

### ステップ4: 解決または続行
- 矛盾が見つかった場合: 確認質問ファイルを生成する
- すべて問題ない場合: 回答をaudit.mdに記録して続行する

---

## 実装ガイドライン

1. **具体的に**: 質問は、どの回答でも明確なアクションにつながるほど正確であること
2. **関連性を保つ**: 現在のステージに必要な質問のみを行う
3. **コンテキストを提供する**: 各質問で回答がなぜ重要かを説明する
4. **質問数を制限する**: 1つの質問ファイルにつき7問以内
5. **論理的に順序付ける**: 一般的な質問から具体的な質問へ流れるようにする

## エラーハンドリング

- ユーザーがレターコードではなく自由記述で回答した場合: 意図を解析し解釈を確認する
- ユーザーが質問をスキップした場合: 未回答項目を強調し、なぜ必要かを説明する
- ユーザーが矛盾する回答を提供した場合: 確認質問ファイルを即座に生成する
