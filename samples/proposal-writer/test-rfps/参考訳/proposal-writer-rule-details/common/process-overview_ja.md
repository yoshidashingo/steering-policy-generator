# 提案書ライター プロセス概要

## 目的

このファイルは提案書作成ワークフローの高レベル概要を提供し、フェーズ間の接続方法、ステージ間の意思決定の流れ、適応型実行モデルの仕組みを説明します。

---

## ワークフローアーキテクチャ

提案書ライターは、適応的なステージ実行を伴う4フェーズの順次プロセスに従います：

### フェーズ1: 分析（ANALYSIS）
**目標**: 何も書く前にRFPを完全に理解する。

分析フェーズは要件の見落としを防ぎ、コンプライアンスのベースラインを確立します。分析をスキップするとコンプライアンス不適合の提案書につながるため、3つのステージすべてが必須です。

**入力**: ユーザーからのRFP文書
**出力**: RFP要約、分類された要件、コンプライアンスマトリクス

### フェーズ2: 計画（PLANNING）
**目標**: コンテンツを起草する前に応答戦略を設計する。

計画フェーズはRFP要件を構造化された提案書計画に変換します。随意契約では競争差別化が不要なため、勝利戦略は条件付きです。

**入力**: 分析フェーズの成果物
**出力**: 提案書構成、勝利戦略（該当する場合）、コンテンツ計画

### フェーズ3: 執筆（WRITING）
**目標**: 説得力のあるコンプライアンス準拠の提案書コンテンツを作成する。

執筆フェーズは実際の提案書セクションを生成します。すべてのRFPがそれらを要求するわけではないため、マネジメント提案と価格構成フレームワークは条件付きです。

**入力**: 計画フェーズの成果物 + ユーザー提供コンテンツ（能力、過去の実績、価格データ）
**出力**: 完全な提案書ドラフトセクション

### フェーズ4: レビュー（REVIEW）
**目標**: 提出前にコンプライアンスと品質を検証する。

レビューフェーズは提案書がすべてのRFP要件を満たし、一貫した品質を維持し、提出準備ができていることを確認します。レビューされていない提案書は許容できないリスクを伴うため、すべてのステージが必須です。

**入力**: すべての執筆済み提案書セクション + コンプライアンスマトリクス
**出力**: コンプライアンスチェックレポート、品質レビューレポート、最終アセンブル済み提案書

## 適応型実行モデル

### ステージ分類

各ステージには実行分類があります：

**常時実行（ALWAYS）**: ワークフローの毎回実行時に実施
- RFP取り込み、要件抽出、コンプライアンスマトリクス
- 提案書構成設計、コンテンツ計画
- エグゼクティブサマリー、技術提案
- コンプライアンスチェック、品質レビュー、最終アセンブリ

**条件付き実行（CONDITIONAL）**: 条件が満たされた場合のみ実施
- 勝利戦略 — 競争入札時のみ実行
- マネジメント提案 — RFPが組織/人員配置情報を要求する場合に実行
- 価格構成フレームワーク — RFPがコスト/価格提案を要求する場合に実行

### 深度の適応

常時実行のステージも深度を適応させます：
- **最小（Minimal）**: 明確で構造化された要件を持つシンプルなRFP
- **標準（Standard）**: 複数の評価領域を持つ中程度の複雑さ
- **包括的（Comprehensive）**: 相互参照された要件、複数巻の複雑なRFP

### 意思決定ポイント

すべてのステージ完了時に、ユーザーが選択します：
- **A) 変更を依頼** — 現在のステージ出力を修正する
- **B) 続行** — 次のステージに進む

これにより、ワークフローが体系的に進行する中でユーザーが完全なコントロールを維持できます。

## フェーズ間のデータフロー

```
分析                     計画                     執筆                     レビュー
+----------------+      +----------------+      +----------------+      +----------------+
| RFP要約        |----->| 提案書構成     |----->| エグゼクティブ |----->| コンプライアンス|
| 要件           |----->| 勝利戦略       |----->| サマリー       |----->| チェック       |
| コンプライアンス|----->| コンテンツ計画 |----->| 技術提案       |----->| 品質レビュー   |
| マトリクス     |      +----------------+      | マネジメント   |      | 最終アセンブリ |
+----------------+                              | 提案           |      +----------------+
                                                | 価格構成       |
                                                | フレームワーク |
                                                +----------------+
```

各フェーズは先行する成果物の上に構築されます。後続フェーズは一貫性とトレーサビリティを維持するために先行する出力を参照します。

## 状態管理

### 提案書状態ファイル（`proposal-state.md`）
全体的なワークフロー進捗を追跡：
- 現在のフェーズとステージ
- 完了/スキップされたステージ（タイムスタンプ付き）
- 下された主要な決定
- 条件付きステージの実行結果

### 監査ファイル（`audit.md`）
すべてのインタラクションを記録：
- ユーザー入力（完全、生、要約しない）
- AI応答と実行されたアクション
- ISO 8601形式のタイムスタンプ
- 各エントリのステージコンテキスト

## 実装ガイドライン

1. **常に分析から開始する** — RFPを理解せずに執筆にスキップしない
2. **フェーズの順序を尊重する** — フェーズは順次実行される。先に進まない
3. **ステージ条件を尊重する** — 実行前に条件付きステージの基準を確認する
4. **トレーサビリティを維持する** — すべての提案書セクションはRFP要件に遡れるようにする
5. **すべてを記録する** — すべてのインタラクション後にaudit.mdに追記する
6. **作成前に検証する** — ファイルを書く前にコンテンツ検証ルールをチェックする

## エラーハンドリング

フェーズが完了できない場合（例：RFPが不完全、要件が矛盾している場合）：
1. 具体的な問題を特定する
2. オプションとともに問題をユーザーに提示する
3. 問題をaudit.mdに記録する
4. 問題が解決されるまで進行しない

詳細なエラー回復手順は `common/error-handling.md` を参照。
