# エラーハンドリングと回復手順

## 目的

このファイルは、提案書作成ワークフローの全フェーズにおけるエラーハンドリング手順を定義します。適切なエラーハンドリングにより、作業を失うことなく、またコンプライアンス不適合の提案書を作成することなく、ワークフローが適切に回復できます。

---

## 一般的なエラーハンドリング原則

### エラー発生時
1. **エラーの特定**: 何が問題かを明確に述べる
2. **影響の評価**: エラーがブロッキングか回避策で対応可能かを判断
3. **コミュニケーション**: エラーとオプションについてユーザーに通知
4. **解決策の提示**: 解決または回避のための明確な手順を提供
5. **文書化**: エラーと解決策をaudit.mdに記録

### エラーの重大度レベル

**致命的（Critical）**: ワークフローを続行できない
- RFP文書が読み取りまたは解析できない
- 必要な提案書セクションが生成できない
- ファイル操作を妨げるシステムエラー

**高（High）**: フェーズが計画通りに完了できない
- 必須の質問への不完全な回答
- ユーザーの矛盾する回答
- 先行フェーズからの依存関係の欠落
- コンプライアンスマトリクスが解決不能なギャップを明らかにした

**中（Medium）**: 回避策で続行可能
- オプション成果物の欠落
- 致命的でない検証の失敗
- 部分的な完了が可能
- 軽微なフォーマット問題

**低（Low）**: 進捗をブロックしない軽微な問題
- スタイルの不整合
- オプション情報の欠落
- ブロッキングでない警告

## フェーズ別エラーハンドリング

### 分析フェーズのエラー

**エラー**: RFP文書が読み取れないまたは破損している
- **原因**: サポートされていないフォーマット、エンコーディング問題、破損ファイル
- **解決策**: サポートされる形式（PDF、DOCX、Markdown、プレーンテキスト）でのRFP提供をユーザーに依頼
- **回避策**: ユーザーの口頭説明に基づいて作業を進める

**エラー**: RFPの構造が不明確または非標準
- **原因**: 非公式なRFP、メールベースの依頼、構造化されたセクションがない
- **解決策**: 要件を構造化するための質問を行う
- **回避策**: ユーザー提供のコンテキストから要件を作成する

**エラー**: RFP内の矛盾する要件
- **原因**: 品質の低いRFP、複数の著者、バージョン管理の問題
- **解決策**: 矛盾をフラグ付けし、発注組織への確認をユーザーに依頼
- **進行しない**: 矛盾が文書化され解決アプローチが合意されるまで

**エラー**: 必須要件と任意要件の判別ができない
- **原因**: RFPの言語が曖昧（「should」vs.「shall」vs.「may」）
- **解決策**: 曖昧な要件の分類をユーザーに依頼
- **デフォルト**: コンプライアンスの安全性のため、曖昧な要件は必須として扱う

### 計画フェーズのエラー

**エラー**: ページ制限内で提案書構成を設計できない
- **原因**: 許可されたページ数に対して要件が多すぎる
- **解決策**: トレードオフオプションを提示（高配点の評価領域を優先）
- **回避策**: 付録で補足的な詳細を補う簡潔な形式を提案

**エラー**: 勝利戦略で差別化要因を特定できない
- **原因**: ユーザーが競争情報を提供できない
- **解決策**: 競争的ポジショニングの代わりに顧客中心の価値提案に焦点を当てる
- **回避策**: ゴースト戦略をスキップし、強みベースの差別化に焦点を当てる

### 執筆フェーズのエラー

**エラー**: エグゼクティブサマリーが割り当て長さを超過
- **原因**: 限られたスペースに対してテーマや要件が多すぎる
- **解決策**: 主要な勝利テーマと最高配点の評価基準を優先
- **回避策**: 詳細コンテンツを技術提案に移動し、サマリーは高レベルに保つ

**エラー**: 技術的アプローチの詳細が不足
- **原因**: 方法論、ツール、アプローチに関するユーザーからの入力不足
- **解決策**: 技術的アプローチに関する的を絞った質問を行う
- **進行しない**: 信頼性のある応答に十分な技術的詳細が提供されるまで

### レビューフェーズのエラー

**エラー**: コンプライアンスチェックで未対応の必須要件を発見
- **原因**: 抽出時の要件見落とし、執筆時のセクション省略
- **解決策**: ギャップに対応するため執筆フェーズに戻る
- **回復**: 適切なセクションに不足コンテンツを追加し、コンプライアンスマトリクスを更新

## 回復手順

### 部分的なステージ完了
**シナリオ**: ステージが実行途中で中断された

**回復手順**：
1. ステージ計画ファイルを読み込む（存在する場合）
2. 最後に完了したステップを特定（最後の [x] チェックボックス）
3. 次の未完了ステップから再開
4. すべての先行ステップが実際に完了していることを確認
5. 通常の実行を続行

### 成果物の欠落
**シナリオ**: 先行フェーズからの必要な成果物が欠落

**回復手順**：
1. どの成果物が欠落しているか特定
2. 再生成可能かどうか判断
3. 可能な場合: そのフェーズに戻り成果物を再生成
4. 不可能な場合: 手動で情報を提供するようユーザーに依頼
5. ギャップをaudit.mdに文書化

### ユーザーがステージの再開を希望
**シナリオ**: ユーザーがステージの結果に不満でやり直しを希望

**回復手順**：
1. 再開の確認（現在の出力はアーカイブされる）
2. 既存の成果物をアーカイブ: `{成果物}.backup`
3. `proposal-state.md` のステージステータスをリセット
4. ステージを最初から再実行

## エスカレーションガイドライン

### ユーザーの支援を求めるタイミング

**即座に**：
- 矛盾するまたは曖昧なユーザー入力
- 会社の能力に関する必要な情報の欠落
- ビジネス判断を要する決定（価格設定、チーミング）
- RFPの解釈に関する質問

**解決を試みた後**：
- 同じステップでの繰り返しエラー
- 利用可能な情報を超える複雑な技術要件
- セクション間の一貫性の問題
- 利用可能なツールでのフォーマットの課題

## ログ要件

### エラーログ形式

```markdown
## エラー - [フェーズ/ステージ名]
**タイムスタンプ**: [ISOタイムスタンプ]
**エラー種別**: [致命的/高/中/低]
**説明**: [何が問題だったか]
**原因**: [なぜ発生したか]
**解決**: [どのように解決されたか]
**影響**: [ワークフローへの影響]

---
```

## 予防のベストプラクティス

1. **早期に検証する**: 分析フェーズでRFPの完全性を確認する
2. **頻繁にチェックポイントを設ける**: ステップ完了後すぐに状態とチェックボックスを更新する
3. **明確にコミュニケーションする**: 各ステージで何をしているか、なぜしているかを説明する
4. **質問する**: 仮定せず、曖昧さは即座に明確にする
5. **すべてを文書化する**: すべての決定と変更をaudit.mdに記録する
6. **継続的に相互チェックする**: 執筆中にセクション間の一貫性を確認する
