# 要件抽出 — 分析フェーズ

## 目的

このステージでは、RFP文書からすべての要件を体系的に抽出・分類・整理します。抽出された要件は、コンプライアンス追跡、提案書構成、およびコンテンツ計画の基盤となります。

## 前提条件
- RFP取り込みステージが完了していること
- `proposal-docs/analysis/rfp-summary.md` が存在し、承認済みであること
- 共通ルールがロード済み：`common/content-validation.md`、`common/question-format-guide.md`

## 実行分類
**タイプ**: ALWAYS（常時実行）
要件抽出は常に実行されます。コンプライアンスにはすべての要件の完全な理解が必要なためです。

## 適応的深度
- **最小限**: 明確に番号付けされた要件を持つシンプルなRFP — 直接的な抽出と分類
- **標準**: 複数セクションにわたって要件が散在する中程度のRFP — セクション横断分析、分類、優先度マッピング
- **包括的**: 暗黙の要件、相互参照、規制基準を含む複雑なRFP — 暗示的要件の抽出、規制分解、トレーサビリティマッピングを含む完全な抽出

---

## 実行ステップ

### ステップ1: 明示的要件の抽出
**アクション**: RFPに明示的に記載されているすべての要件を特定する
**入力**: RFP文書全体
**出力**: 以下を含む要件リスト:
- 要件ID（割り当て済みまたはRFPのもの）
- 要件テキスト（RFPからの原文そのまま）
- 出典箇所（セクション番号およびページ）
- 要件タイプ（機能的、技術的、管理的、事務的）
**検証**: RFPのすべてのセクションがレビュー済みであること。明示的要件の見落としがないこと

### ステップ2: 暗黙的要件の抽出
**アクション**: RFPで暗示されているが明示的に記載されていない要件を特定する
**入力**: RFP文書全体、評価基準、参照されている業界標準
**出力**: 以下を含む暗黙的要件リスト:
- 導出された要件テキスト
- 出典（この要件を暗示している根拠）
- 確信度（高 / 中 / 低）
**検証**: 暗黙的要件が合理的な推論であり、仮定ではないこと（確信度「低」の項目はユーザーレビューのためフラグ付け）

### ステップ3: 要件の分類
**アクション**: 各要件を優先度とタイプ別に分類する
**入力**: 抽出されたすべての要件（明示的および暗黙的）
**出力**: 分類された要件:
- **必須**（shall/must）: 準拠しなければ失格のリスクあり
- **望ましい**（should）: 対応すればスコアリング上の価値が加算される
- **情報提供**（may/describe）: 厳格な準拠基準のない、要求される情報
**検証**: 分類がRFPの文言と整合していること。曖昧な文言はユーザー判断のためフラグ付け

### ステップ4: 評価基準マッピング
**アクション**: 要件を評価基準およびスコアリングの重み付けにマッピングする
**入力**: 要件リスト、RFP要約からの評価基準
**出力**: 以下を示す要件-基準マッピング:
- どの要件がどの評価基準に影響するか
- スコアリングの重み付けに基づく相対的重要度
- 重要要件（必須 + 高重み付け評価）
**検証**: すべての評価基準に関連する要件が紐付けられていること。最高重み付けの基準に十分な要件カバレッジがあること

### ステップ5: 提出形式要件
**アクション**: 提案書の形式および提出に関するすべての要件を抽出する
**入力**: RFPの提出指示セクション
**出力**: 以下を含む形式要件:
- ページ制限（全体および指定がある場合はセクション別）
- フォント、余白、フォーマット仕様
- 巻構成（複数巻が必要な場合）
- 必須セクションとその順序
- 添付資料、フォーム、必要な証明書
- 電子提出 vs. 物理提出の詳細
**検証**: すべての形式制約が文書化されていること。コンテンツ要件との矛盾がフラグ付けされていること

### ステップ6: コンプライアンスおよび規制要件
**アクション**: 規制、法的、コンプライアンス要件を特定する
**入力**: RFPの契約条件、参照される基準および規制
**出力**: 以下を含むコンプライアンス要件:
- 参照基準（ISO、NIST、CMMIなど）
- 規制準拠（HIPAA、FISMA、GDPRなど）
- 認証要件
- 保険および保証要件
- 中小企業/多様性要件
**検証**: すべての参照基準が特定されていること。自組織に適用されるものをユーザーが確認済みであること

### ステップ7: 曖昧性と矛盾の特定
**アクション**: 曖昧、矛盾、または不完全な要件をフラグ付けする
**入力**: 抽出されたすべての要件
**出力**: 以下を含む問題リスト:
- 曖昧な要件（複数の妥当な解釈が可能）
- 矛盾する要件（両方を完全に満たすことができない2つの要件）
- 不完全な要件（回答に必要な詳細が不足）
- 各項目への推奨（発行者に確認、解釈を仮定、ユーザー判断を要請）
**検証**: フラグ付けされたすべての項目がユーザーに解決のため提示されること

### ステップ8: 要件文書の生成
**アクション**: 包括的な要件文書を作成する
**入力**: ステップ1〜7のすべての出力
**出力**: `common/output-structure-patterns.md` の要件文書パターンに従った `proposal-docs/analysis/requirements.md`
**検証**: 文書が適切な分類とトレーサビリティを持つすべての抽出要件を含んでいること

## 質問の生成

### 質問カテゴリ
- **分類**: 要件が必須か望ましいかに関する質問
- **解釈**: 曖昧な要件の文言をどう解釈するかに関する質問
- **能力**: 特定の要件への準拠能力に関する質問
- **優先度**: トレードオフが必要な場合にどの要件を重視するかに関する質問

### 質問ファイルテンプレート

```markdown
# 要件抽出に関する質問

抽出された要件を明確にするため、以下の質問にお答えください。

## 質問 1
RFPのセクション [X] に「[曖昧なテキスト]」と記載されています。これは以下のように解釈できます:

A) [解釈1 — より制約的]
B) [解釈2 — より緩和的]
C) [解釈3 — 別の読み方]
D) その他（以下の [Answer]: タグの後に記述してください）

[Answer]:
```

## エラー処理

**エラー**: 要件が必須か望ましいか判断できない
- **解決策**: ユーザーに要件の分類を尋ねる質問ファイルを生成する
- **デフォルト**: コンプライアンスの安全性のため必須として扱う

**エラー**: RFPが文書に含まれていない外部基準を参照している
- **解決策**: 参照をフラグ付けし、ユーザーに基準文書を持っているか確認する
- **回避策**: 要件に「参照基準 — コンプライアンスTBD」として記録する

**エラー**: 要件が互いに矛盾している
- **解決策**: 両方の要件を文書化し、矛盾をフラグ付けし、ユーザーに解決方針を確認する
- **進行禁止**: 矛盾する要件が未解決のままコンプライアンスマトリクスに進まないこと

一般的なエラー処理手順については `common/error-handling.md` を参照してください。
