# 技術レスポンス — 執筆フェーズ

## 目的

このステージでは、技術レスポンスセクションを作成します。これは通常、評価において最も配点の高いコンポーネントです。技術レスポンスは、提案する作業がどのように（HOW）実施されるかを示し、評価者がアプローチの実現可能性、品質、革新性を評価するための十分な詳細を提供します。

## 前提条件
- 分析フェーズおよび計画フェーズが完了済み
- エグゼクティブサマリーが完了済み
- すべての先行成果物がロード済み：
  - `proposal-docs/analysis/requirements.md`（技術要件）
  - `proposal-docs/analysis/compliance-matrix.md`（技術コンプライアンス状況）
  - `proposal-docs/planning/proposal-structure.md`（技術セクション構成）
  - `proposal-docs/planning/content-plan.md`（技術セクションのアウトライン）
  - `proposal-docs/planning/win-strategy.md`（技術的差別化要因、存在する場合）
  - `proposal-docs/proposal/executive-summary.md`（整合性確認用）
- 共通ルールがロード済み：`common/content-validation.md`、`common/overconfidence-prevention.md`

## 実行分類
**タイプ**：常時実行（ALWAYS）
技術レスポンスは常に実行されます。すべての提案書は提案アプローチを記述する必要があるためです。

## 適応的深度
- **最小（Minimal）**：シンプルなRFP — 成果物とタイムラインを含む簡潔な方法論の説明
- **標準（Standard）**：中程度のRFP — アーキテクチャ、方法論、ツール、リスク軽減策、品質保証を含む詳細なアプローチ
- **包括的（Comprehensive）**：複雑なRFP — 詳細なアーキテクチャ図、実施フェーズ、人員配置モデル、リスクマトリクス、品質フレームワークを含む深い技術的内容

## 実行ステップ

### ステップ1：技術レスポンスのアウトライン作成（パート1 — 計画）
**アクション**：ユーザー承認用の詳細な技術レスポンスのアウトラインを作成する
**入力**：コンテンツプラン（技術セクション）、要件、評価基準
**出力**：以下を含むアウトライン：
- 技術アプローチ／方法論概要の構成
- ソリューションアーキテクチャ／システム設計の構成
- 実施計画の構成（フェーズ、マイルストーン、タイムライン）
- リスク特定と軽減策の構成
- 品質保証アプローチの構成
- 技術スタックとツールの構成
**検証**：アウトラインがすべての技術評価基準に対応していること。構成がRFP要件（またはRFP指定フォーマット）に一致していること

### ステップ2：アウトラインのユーザー承認
**アクション**：アウトラインを提示し、技術アプローチに関するユーザーの意見を収集する
**入力**：ステップ1のアウトライン
**出力**：ユーザーが提供した技術的詳細を含む承認済みアウトライン
**検証**：ドラフト作成前にユーザーが技術アプローチの方向性を確認していること

### ステップ3：技術アプローチ／方法論のドラフト作成（パート2 — 生成）
**アクション**：全体的な方法論を説明する技術アプローチセクションを執筆する
**入力**：コンテンツプラン、ユーザー提供の方法論詳細、勝利テーマ
**出力**：以下を含む技術アプローチセクション：
- 方法論の概要（フレームワーク、ライフサイクル、プロセス）
- テーラリングアプローチ（方法論をこの特定プロジェクトにどう適応させるか）
- 主要な原則とプラクティス
- 顧客が掲げる目標との整合
**検証**：方法論が具体的であること（汎用的でない）。ユーザーの実際のプラクティスと整合していること。関連する勝利テーマが統合されていること

### ステップ4：ソリューションアーキテクチャ／システム設計のドラフト作成
**アクション**：ソリューションアーキテクチャまたはシステム設計セクションを執筆する
**入力**：技術要件、コンテンツプラン、ユーザーの入力
**出力**：以下を含むアーキテクチャセクション：
- システムアーキテクチャの概要（図の説明を含む）
- コンポーネントの説明と相互作用
- 根拠を伴った技術選定
- 既存システムとの統合ポイント（該当する場合）
- スケーラビリティとパフォーマンスの考慮事項
**検証**：アーキテクチャがすべての機能要件および技術要件に対応していること。技術選定が最新かつ適切であること

### ステップ5：実施計画のドラフト作成
**アクション**：フェーズ、マイルストーン、タイムラインを含む実施計画を執筆する
**入力**：要件（履行期間、マイルストーン）、コンテンツプラン
**出力**：以下を含む実施計画：
- フェーズごとの目標と成果物を含むフェーズ分割
- 主要なマイルストーンと意思決定ポイント
- タイムライン（ガントチャートの説明またはマイルストーン表）
- 依存関係とクリティカルパス
- 移行／オンボーディング計画（既存事業者から引き継ぐ場合）
**検証**：タイムラインがRFPの履行期間内に収まっていること。マイルストーンが現実的であること。依存関係が論理的であること

### ステップ6：リスク軽減策のドラフト作成
**アクション**：リスク特定と軽減策セクションを執筆する
**入力**：技術アプローチ、コンプライアンスマトリクス（ギャップ）、実施計画
**出力**：以下を含むリスク軽減策セクション：
- 特定された主要リスク（技術、スケジュール、リソース、統合）
- リスクの発生確率と影響度の評価
- 特定された各リスクに対する軽減戦略
- リスクの監視とエスカレーション手順
**検証**：リスクが現実的であること（汎用的でない）。軽減戦略が具体的かつ実行可能であること

### ステップ7：品質保証アプローチのドラフト作成
**アクション**：品質保証セクションを執筆する
**入力**：要件（品質基準、テスト要件）、方法論
**出力**：以下を含むQAセクション：
- 品質管理フレームワーク
- テスト戦略（単体、結合、システム、受入）
- 品質指標と受入基準
- 継続的改善プロセス
- 標準規格への準拠（ISO、CMMIなど、該当する場合）
**検証**：QAアプローチがRFPの品質要件と整合していること。標準規格が要件ドキュメントに合致していること

### ステップ8：技術レスポンスの組み立てと検証
**アクション**：すべてのセクションを統合し、一貫性のある技術レスポンスにまとめる
**入力**：ステップ3〜7のすべてのドラフトセクション
**出力**：`proposal-docs/proposal/technical-response.md` に完成した技術レスポンス
**検証**：
- 合計の長さがページ予算内であること
- コンプライアンスマトリクスのすべての技術要件に対応していること
- エグゼクティブサマリーとの整合性が取れていること
- 技術用語と製品名に一貫性があること
- 相互参照が正確であること
- content-validation.md に基づくコンテンツ検証を実施済みであること

## エラーハンドリング

**エラー**：ユーザーからの入力が不十分で技術アプローチの具体性が欠けている
- **解決策**：ユーザー向けの具体的な技術質問を生成する。汎用的な内容のまま進めない
- **回避策**：利用可能な詳細レベルで記述し、ユーザー入力が必要な箇所を「[要詳細（DETAIL NEEDED）]」と明示する

**エラー**：アーキテクチャセクションでテキストでは表現できない図が必要
- **解決策**：図を詳細に記述する。レビュー用にASCIIまたはMermaid形式の表現を提供する
- **注記**：最終的なグラフィックスはユーザーが外部で作成する必要がある場合がある

**エラー**：実施タイムラインがRFPの履行期間に収まらない
- **解決策**：フェーズ構成を見直す。並行活動を検討する。トレードオフの選択肢をユーザーに提示する
- **提出してはならないもの**：RFPの指定する履行期間と矛盾するタイムライン

一般的なエラーハンドリング手順については `common/error-handling.md` を参照してください。
