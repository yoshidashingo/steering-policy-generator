# コンテンツ検証ルール

## 目的

このファイルは、提案書作成ワークフロー中にファイルを作成する前に必ず適用する検証ルールを定義します。コンテンツ検証は、フォーマットエラー、壊れた参照、不整合を防ぎ、提案書の品質を保護します。

---

## 作成前検証チェックリスト

提案書成果物を作成する前に以下を確認：

### 1. 文書構造の検証
- [ ] すべての見出しが正しい階層に従っている（H1 > H2 > H3、レベルのスキップなし）
- [ ] 目次エントリが実際のセクション見出しと一致している
- [ ] セクション番号が連続的かつ一貫している
- [ ] 孤立したセクション（コンテンツのないセクション）がない

### 2. 相互参照の検証
- [ ] すべての内部相互参照が既存のセクションを指している
- [ ] コンプライアンスマトリクスの参照が実際の提案書セクションにマッピングされている
- [ ] 提案書内の要件IDがRFP分析の要件IDと一致している
- [ ] 図表番号が連続的で本文中で参照されている

### 3. フォーマットの検証
- [ ] 全体を通じて一貫した見出しの大文字化スタイル
- [ ] セクション内で一貫したリストフォーマット（箇条書き vs. 番号）
- [ ] テーブルにヘッダーがあり、一貫した列揃え
- [ ] 同一ファイル内でインデントスタイルが混在していない

### 4. コンテンツ整合性の検証
- [ ] プレースホルダーテキストが残っていない（例：「[TBD]」「[ここに挿入]」）
- [ ] セクション間で矛盾する記述がない
- [ ] 技術用語が全体を通じて一貫して使用されている
- [ ] 略語が各主要セクションの初出時に定義されている

### 5. 特殊文字の処理
- [ ] Markdownの特殊文字がリテラルで使用される場合に適切にエスケープされている
- [ ] 通貨記号と数値が一貫してフォーマットされている
- [ ] 日付形式が一貫している（内部文書ではISO 8601推奨）
- [ ] パーセンテージと単位の形式が一貫している

## 提案書固有の検証ルール

### コンプライアンスマトリクスの検証
- すべてのRFP要件IDがコンプライアンスマトリクスに出現すること
- すべてのコンプライアンスマトリクスエントリにステータスがあること（準拠/一部準拠/非準拠/該当なし）
- 一部準拠と非準拠のエントリに緩和策の注記があること
- マトリクスが各要件の対応箇所となる提案書セクションを相互参照していること

### 要件トレーサビリティの検証
- 抽出されたすべての要件がRFP内のソース位置に遡れること
- すべての提案書セクションが少なくとも1つのRFP要件に遡れること
- 対応するRFPの根拠なしに提案書セクションが存在しないこと
- 前方および後方のトレーサビリティが完全であること

### セクション長の検証
- RFPがページ制限を指定している場合、セクションごとの推定ページ数を追跡する
- 割り当てられたページ予算を超えるセクションをフラグ付けする
- 重要なセクションが十分なページ割り当てを受けていることを確認する
- 全体制限に対する合計ページ数を追跡する

### セクション間の一貫性検証
- 会社名とブランディングがすべてのセクションで同一であること
- チームメンバーの名前と肩書が一貫していること
- 技術用語と製品名が全体を通じて一致していること
- プロジェクトタイムラインの日付がセクション間で一貫していること

## 検証プロセス

### 検証のタイミング
1. 新しいファイルを**作成する前**
2. 各提案書セクションの**完了後**
3. レビューフェーズの**コンプライアンスチェック中**
4. 完全な提案書の**最終アセンブリ前**

### 検証の重大度レベル

**ブロッキング** — 続行前に修正必須：
- 壊れた相互参照
- 必須セクションの欠落
- 必須要件のコンプライアンスマトリクスギャップ
- セクション間の矛盾するコンテンツ

**警告** — 修正すべきだが、承認を得て続行可能：
- 軽微なフォーマットの不整合
- 任意セクションのギャップ
- 推定ページ予算の超過（10%以内）

**情報** — 修正するのが望ましいが、進捗をブロックしない：
- スタイルの好み（例：オックスフォードカンマの使用）
- 任意セクションの拡充
- 見た目のフォーマットの違い

## 実装ガイドライン

1. **段階的に検証する**: 最後だけでなく、各セクション作成時にチェックする
2. **検証結果を記録する**: 検証結果をaudit.mdに記録する
3. **ブロッキング問題は即座に修正する**: ブロッキング検証の失敗がある場合は進行しない
4. **警告を集約する**: フェーズ完了時にすべての警告を提示し一括解決する
5. **コンプライアンススコアを維持する**: 対応済み要件のパーセンテージを追跡する

## エラーハンドリング

- 検証で矛盾が判明した場合: フラグを立て、続行前にユーザーに解決を依頼する
- 相互参照が壊れている場合: ソースとターゲットを特定し、修正を提案する
- ページ制限を超えた場合: 影響評価付きのコンテンツ削減オプションを提示する
- 一般的なエラーハンドリング手順は `common/error-handling.md` を参照
