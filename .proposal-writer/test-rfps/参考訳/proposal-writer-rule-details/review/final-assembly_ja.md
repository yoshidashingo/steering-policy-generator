# 最終組立 — レビューフェーズ

## 目的

このステージでは、承認済みのすべての提案書セクションを最終提出パッケージにまとめ、目次を生成し、提出チェックリストを作成し、最終的な提案書サマリーを作成します。これは提案書が提出可能になる前の最後のステージです。

## 前提条件
- コンプライアンスチェックが完了していること（すべての重大なギャップが解決済み）
- 品質レビューが完了していること（重大な改善が対処済み）
- `proposal-docs/review/compliance-check-report.md` — 最終コンプライアンスステータス
- `proposal-docs/review/quality-review-report.md` — 最終品質ステータス
- 承認済みのすべての提案書セクション：
  - `proposal-docs/proposal/executive-summary.md`
  - `proposal-docs/proposal/technical-response.md`
  - `proposal-docs/proposal/management-response.md`（実行された場合）
  - `proposal-docs/proposal/pricing-framework.md`（実行された場合）
- 共通ルールの読み込み：`common/content-validation.md`、`common/output-structure-patterns.md`

## 実行分類
**タイプ**: 常時実行（ALWAYS）
最終組立は常に実行されます。すべての提案書は統一された提出パッケージにまとめる必要があるためです。

---

## 実行ステップ

### ステップ1：最終コンテンツインベントリ
**アクション**: 必要なすべての提案書コンポーネントが存在し、承認されていることを検証する
**入力**: 提案書構成文書、すべての執筆済みセクション
**出力**: コンテンツインベントリ：
- すべてのセクションのリストとステータス（存在 / 欠落 / 下書き）
- セクションごとのページ数
- 総ページ数と制限の対比
- すべての必要な添付書類および書類の特定
**検証**: すべての必要なセクションが存在すること。「下書き」ステータスのセクションがないこと。ページ数が制限内であること

### ステップ2：目次の生成
**アクション**: 正確なセクション参照を含む最終目次を生成する
**入力**: 最終順序のすべての提案書セクション
**出力**: 以下を含む完全な目次：
- すべてのセクションおよびサブセクションの見出し
- ページ参照（コンテンツの長さに基づく推定）
- 図表一覧（該当する場合）
**検証**: 提案書内のすべての見出しが目次に記載されていること。ページ参照が連続的かつ妥当であること

### ステップ3：クロスリファレンスの検証
**アクション**: すべての内部クロスリファレンスが正しく解決されることを検証する
**入力**: すべての提案書セクション
**出力**: クロスリファレンスの検証結果：
- すべての「セクションXを参照」の参照先が存在すること
- コンプライアンスマトリクスのセクション参照が正確であること
- 図表参照が正しいこと
- 壊れたリファレンスや循環リファレンスがないこと
**検証**: クロスリファレンスの100%が解決されること。孤立したリファレンスがないこと

### ステップ4：最終フォーマットチェック
**アクション**: すべてのセクション間でフォーマットが一貫していることを検証する
**入力**: すべての提案書セクション
**出力**: フォーマット検証：
- 見出しスタイルがセクション間で一貫していること
- リストのフォーマットが一貫していること（箇条書き、番号付き）
- 表のフォーマットが一貫していること
- 用語および略語の使用が一貫していること
- プレースホルダーテキストが残っていないこと（「[TBD]」「[INSERT]」「[USER INPUT]」）
**検証**: フォーマットの不整合がないこと。プレースホルダーテキストがないこと（またはユーザー依存フィールドとして明確にマークされていること）

### ステップ5：提出チェックリストの生成
**アクション**: 納品前の最終検証のための提出チェックリストを作成する
**入力**: RFP提出要件、コンテンツインベントリ
**出力**: `common/output-structure-patterns.md` の提出チェックリストパターンに従った `proposal-docs/review/submission-checklist.md`
**検証**: チェックリストがすべてのRFP提出要件をカバーしていること。漏れがないこと

### ステップ6：最終提案書サマリー
**アクション**: 参照用に完成した提案書のサマリーを作成する
**入力**: すべての提案書コンポーネント、コンプライアンスチェックレポート、品質レビューレポート
**出力**: 以下を含む最終サマリー：
- 提案書の統計情報（セクション数、ページ数、対応済み要件数）
- コンプライアンススコア（完全対応済み要件の割合）
- 品質評価（品質レビューから）
- 主要な勝利テーマ
- 条件付きステージの実行/スキップ状況
- ユーザーの注意が必要な未解決項目
**検証**: サマリーが最終的な提案書の状態を正確に反映していること

### ステップ7：最終状態の更新
**アクション**: 完了を反映するために proposal-state.md を更新する
**入力**: すべての完了データ
**出力**: すべてのステージが完了としてマークされた `proposal-docs/proposal-state.md` の更新
**検証**: 状態ファイルがワークフローの完了を正確に反映していること

## エラーハンドリング

**エラー**: 必要なセクションが欠落しているか不完全な場合
- **解決策**: 欠落しているセクションを特定する。該当する執筆フェーズのステージに戻る
- **組み立てないこと**: 不完全な提案書は組み立てないこと（すべての必要なセクションが存在する必要がある）

**エラー**: ページ数が制限を超えている場合
- **解決策**: セクションごとのページ超過を提示する。削減の優先順位を推奨する
- **復旧方法**: 優先度の低いコンテンツを削減する。冗長な内容を凝縮する。付録の活用を検討する

**エラー**: プレースホルダーテキストが提案書に残っている場合
- **解決策**: 残っているすべてのプレースホルダーとその場所をリスト化する
- **注意**: 一部のプレースホルダーは意図的なもの（価格などのユーザー依存フィールド）である可能性がある。エラーと意図的な空欄を区別すること

**エラー**: 最終編集後にクロスリファレンスが壊れている場合
- **解決策**: クロスリファレンスの検証を再実行する。壊れたリファレンスを修正する
- **復旧方法**: 最終構成を反映するようにセクション参照を更新する

一般的なエラーハンドリング手順については `common/error-handling.md` を参照してください。
