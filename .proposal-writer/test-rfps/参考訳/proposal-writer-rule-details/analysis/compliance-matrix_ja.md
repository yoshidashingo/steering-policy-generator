# コンプライアンスマトリクス — 分析フェーズ

## 目的

このステージでは、RFPのすべての要件に対する組織の準拠能力を追跡する包括的なコンプライアンスマトリクスを作成します。コンプライアンスマトリクスは提案書プロセス全体を通じた中心的な追跡文書であり、後続の各ステージで更新されます。

## 前提条件
- RFP取り込みおよび要件抽出ステージが完了していること
- `proposal-docs/analysis/rfp-summary.md` が存在し、承認済みであること
- `proposal-docs/analysis/requirements.md` が存在し、承認済みであること
- 共通ルールがロード済み：`common/content-validation.md`、`common/question-format-guide.md`

## 実行分類
**タイプ**: ALWAYS（常時実行）
コンプライアンスマトリクスは常に実行されます。すべての提案書はRFP要件への準拠を示す必要があるためです。

## 適応的深度
- **最小限**: 要件数が少ないシンプルなRFP — 基本的な準拠/非準拠ステータス
- **標準**: 中程度のRFP — ギャップに対する緩和策メモを含む完全な準拠ステータス
- **包括的**: 複雑なRFP — リスクスコアリング、緩和戦略、証拠マッピングを含む詳細なコンプライアンス分析

## 実行ステップ

### ステップ1: コンプライアンスマトリクスの初期化
**アクション**: 抽出されたすべての要件を含むコンプライアンスマトリクス構造を作成する
**入力**: `proposal-docs/analysis/requirements.md`
**出力**: すべての要件ID、説明、および空のステータスフィールドを含む初期マトリクス
**検証**: 要件文書のすべての要件がマトリクスに含まれていること

### ステップ2: コンプライアンス評価
**アクション**: 各要件の初期準拠ステータスを評価する
**入力**: 要件リスト、ユーザー提供の能力情報
**出力**: 各要件について以下のいずれか:
- **準拠**: 要件を完全に満たすことができる
- **部分的準拠**: 制限、代替策、または条件付きで満たすことができる
- **非準拠**: 要件を満たすことができない
- **該当なし**: この回答にはこの要件が適用されない
- **TBD**: 追加情報なしでは準拠を判断できない
**検証**: 未評価の要件がないこと。TBD項目には具体的なフォローアップ質問があること

### ステップ3: ギャップ分析
**アクション**: コンプライアンスギャップ（部分的準拠、非準拠、TBD項目）を分析する
**入力**: ステップ2のコンプライアンス評価
**出力**: 各ギャップについて:
- 影響評価（評価スコアリングにどの程度影響するか？）
- リスクレベル（高 / 中 / 低）
- 根本原因（能力ギャップ、情報ギャップ、スコープギャップ）
**検証**: すべてのギャップに影響評価とリスク評価があること

### ステップ4: 緩和戦略の策定
**アクション**: コンプライアンスギャップに対する緩和戦略を策定する
**入力**: ステップ3のギャップ分析
**出力**: 各ギャップについて以下の1つ以上:
- **代替アプローチ**: 要件の意図を満たす別の方法
- **根拠付き例外**: 理由の説明と代替策を伴う正式な例外申請
- **開発計画**: タイムラインを伴う能力開発のコミットメント
- **チーミング戦略**: 該当能力を持つ他の組織との提携
- **リスク受容**: ギャップを認識し、利用可能な最善の回答を提供する
**検証**: すべての部分的準拠および非準拠の要件に少なくとも1つの緩和戦略があること

### ステップ5: 提案書セクションマッピング
**アクション**: 各要件を対応する提案書セクションにマッピングする
**入力**: コンプライアンスマトリクス、提案書構成案（RFPの推奨形式に基づく）
**出力**: 以下を含む更新済みマトリクス:
- 各要件の対象提案書セクション
- 提案書セクション別にグループ化された要件
- 複数セクションにまたがる要件の相互参照計画
**検証**: すべての準拠および部分的準拠の要件が少なくとも1つの提案書セクションにマッピングされていること

### ステップ6: リスクスコアリング
**アクション**: 全体的なコンプライアンスリスクスコアを算出する
**入力**: 評価と緩和策を含む完成コンプライアンスマトリクス
**出力**: 以下を含むリスク要約:
- 全体的な準拠率
- 高リスク要件の数
- 重大ギャップ（非準拠またはTBDステータスの必須要件）
- 評価基準別リスクスコア
**検証**: リスクスコアがギャップ分析と整合していること。重大ギャップが目立つようにフラグ付けされていること

### ステップ7: コンプライアンスマトリクス文書の生成
**アクション**: 最終的なコンプライアンスマトリクス文書を生成する
**入力**: ステップ1〜6のすべての出力
**出力**: `common/output-structure-patterns.md` のコンプライアンスマトリクスパターンに従った `proposal-docs/analysis/compliance-matrix.md`
**検証**: マトリクスが完全で、内部的に整合しており、相互参照が正確であること

## 質問の生成

### 質問カテゴリ
- **能力**: 組織が特定の要件を満たせるかどうかに関する質問
- **緩和策**: コンプライアンスギャップに対する望ましいアプローチに関する質問
- **優先順位**: 限られたリソースの中でどのギャップを優先的に対処するかに関する質問
- **例外**: 特定の要件に対して例外を申請するかどうかに関する質問

### 質問ファイルテンプレート

```markdown
# コンプライアンスマトリクスに関する質問

RFP要件へのコンプライアンスを評価するため、以下の質問にお答えください。

## 質問 1
要件 [ID] は「[要件の説明]」を求めています。貴組織は完全に準拠できますか？

A) はい、完全に準拠 — この要件を完全に満たします
B) 部分的に — 以下の制限付きでこの要件を満たせます: [Answerタグの後に記述してください]
C) いいえ — この要件を満たすことができません
D) その他（以下の [Answer]: タグの後に記述してください）

[Answer]:
```

## エラー処理

**エラー**: 追加の能力情報なしではコンプライアンスを判断できない
- **解決策**: 特定の能力ギャップを対象とした質問ファイルを生成する
- **回避策**: TBDとしてマークし、計画フェーズでの解決を追跡する

**エラー**: 必須要件が非準拠であり、実行可能な緩和策がない
- **解決策**: ユーザーとGo/No-Go判断を協議する — このRFPが適切でない可能性を示唆
- **エスカレーション**: これはユーザーの判断を要するビジネス上の意思決定である

**エラー**: 要件が曖昧すぎてコンプライアンスを評価できない
- **解決策**: TBDとして文書化し、ユーザーに発行組織への確認を推奨する
- **回避策**: 最も可能性の高い解釈に基づいて評価し、仮定としてフラグ付けする

一般的なエラー処理手順については `common/error-handling.md` を参照してください。
